---

## 1. 核心状态定义

Bot 以**状态机**运行，任何时刻处于以下四个状态之一。**「闲聊」是默认常驻态**，其余三态执行完即回到闲聊。

| 状态 | 名称 | 职责 |
| --- | --- | --- |
| 状态 1 | 引入 Onboarding | 用户首次进入，告知功能与操作。**不强迫选城市**，仅做引导。 |
| 状态 2 | 闲聊 Chitchat | **默认常驻状态**。负责语义判断与人设对话，是所有流转的中枢。 |
| 状态 3 | 导入 Import | 功能执行状态。处理链接解析与图片识别，解析存库后回到闲聊。 |
| 状态 4 | 交付 Delivery | 结果产出状态。发送 H5 决策页链接，发送后回到闲聊。 |

**状态流转规则**：状态 3、4 是「执行完即返回」的临时态，做完动作必定回到状态 2。状态 2 是中枢，所有意图判断都在这里发生。

---

## 2. 状态 2（闲聊）下的决策树

每收到一条消息，Bot 优先做**语义判断**，归入下面三个分支之一。

### 分支 A — 识别为「导入酒店」意图
触发信号：发了链接 / 图片 / 提到某酒店。

```
是否已选城市？
├─ 否 → 触发引导文案，告知需先选城市（不进入导入）
└─ 是 → 进入【状态 3 导入】→ 解析并存库 → 完成后回到【状态 2】
```

### 分支 B — 识别为「完成导入」意图
触发信号：如「好了」「看结果」「没了」。

```
判断 1：是否已选城市？
├─ 否 → 触发引导文案，告知需先选城市
└─ 是 → 进入判断 2

判断 2：是否已有导入酒店？
├─ 否 → 告知目前还没酒店，建议发送截图或链接
└─ 是 → 进入【状态 4 交付】→ 发送 H5 链接 → 完成后回到【状态 2】
```

### 分支 C — 识别为「普通对话」意图
触发信号：非导入、非完成的任何消息。

```
执行：以「向导人设」风趣回复 → 维持在【状态 2】
```

## 3. 逻辑关键点对齐

| 关键点 | 说明 |
| --- | --- |
| **非强制性** | 用户在「引入」后拥有高度自由：可以先闲聊，也可以直接喂数据，不被流程绑死。 |
| **前置依赖** | 所有功能执行（导入、看结果）都**强依赖于「已选城市」**——城市是地图上计算通勤的锚点。 |
| **容错弥补** | 在「看结果」阶段若缺数据（城市或酒店），Bot 以**积极态度**指出缺失项并引导补全，不是冷冰冰报错。 |

## 4. 云端 AI 实现链路

收到消息后，Bot **不直接把整段对话丢给大模型**，而是先**判断意图**拿到一个 `prompt_key`，用 key 取出对应的 Prompt 模板，再把「Prompt + 上下文」一起发给 DeepSeek API。

```
收到消息
  → 判断模块（意图识别）→ 得到 prompt_key
  → 用 prompt_key 取出对应 Prompt 模板
  → 组装 [Prompt 模板 + 上下文]
  → 调用 DeepSeek API
  → 回复用户
```

**判断模块是核心**：它把「自然语言意图」收敛成有限个 `prompt_key`，每个 key 对应一段可单独维护、可灰度调整的 Prompt，使大模型行为**可控、可测、可迭代**。

### 伪代码

```python
def handle_message(msg, ctx):
    prompt_key = classify_intent(msg, ctx)        # 判断模块 → prompt_key
    prompt = PROMPT_TABLE[prompt_key]             # 取模板
    reply = deepseek(prompt=prompt, context=ctx)  # Prompt + 上下文 → DeepSeek
    return reply
```

## 5. Prompt 表（草案 · 具体 Prompt 待填）

每个场景对应一个 `prompt_key` 与一段 Prompt 模板。下表为场景清单与设计理由，`prompt_key` 命名建议作为后续实现的字典键。

| # | prompt_key（建议） | 需求场景 | 设计 / 修改理由 |
| --- | --- | --- | --- |
| 1 | `greeting_guide` | 问候与上手指南 | 不只是打招呼，要立刻教用户怎么「发卡片 / 截图」。 |
| 2 | `init_city_setup` | 初始化引导（选城市 / 景点） | **新增**。确保地图上有计算通勤的「锚点」。 |
| 3 | `import_processing` | 数据输入实时反馈（处理中） | **合并**。体现「正在扫描 / 解析」的进度感。 |
| 4 | `import_result` | 解析结果反馈（含纠错入口） | **强化**。成功报喜；失败体现「积极弥补」态度并引导重发。 |
| 5 | `compare_nudge` | 对比决策触发（Nudge） | **新增**。当数据足够时，推用户去网页看结果。 |
| 6 | `query_detail` | 咨询地点 / 酒店详情 | 保持现状，体现「整合者」的专业。 |
| 7 | `h5_entry` | 返回规划网址（H5 入口） | 核心入口，文案要诱人。 |
| 8 | `boundary_paywall` | 能力边界与付费引导 | **合并**。把「我不能订房」和「请付费开启高级工具」逻辑连贯起来。 |
| 9 | `billing` | 计费相关（支付 / 到期） | 保持现状，确保商业闭环。 |
| 10 | `chitchat_persona` | 向导式闲聊（含认错弥补） | **强化人设**。体现「可靠但会犯错」的亲切感。 |
| 11 | `share_feedback` | 分享与反馈引导 | 增加传播属性。 |
