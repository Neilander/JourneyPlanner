# 云端 AI Bot 设计

> JourneyPlanner 客服机器人的设计文档。给 AI / 工程实现阅读。
> 内容：状态机（5 态）→ 三个新增模块 → 闲聊决策树 → 删除酒店子状态机 → AI 实现链路 → Prompt 表。

---

## 1. 核心状态定义

Bot 以**状态机**运行，任何时刻处于以下五个状态之一。**「闲聊」是默认常驻态（idle）**，是所有意图判断的中枢；其余四态执行完即回到闲聊。

| 状态 | 名称 | 职责 |
| --- | --- | --- |
| 状态 1 | 引入 Onboarding | 用户首次进入，告知功能与操作。**不强迫选城市**，仅做引导。**承载「功能介绍模块」**。 |
| 状态 2 | 闲聊 Chitchat（idle） | **默认常驻状态**。负责语义判断与人设对话，是所有流转的中枢。 |
| 状态 3 | 导入 Import | 功能执行状态。处理链接解析与图片识别，解析存库后回到闲聊。 |
| 状态 4 | 交付 Delivery | 结果产出状态。发送 H5 决策页链接，发送后回到闲聊。 |
| 状态 5 | 删除 Delete | 功能执行状态。把用户想删的酒店从已导入列表中**带确认地**移除。含 2 个子状态。 |

**状态流转规则**：状态 3、4、5 都是「执行完即返回 idle」的功能态。状态 2 是中枢，所有意图判断都在这里发生。

---

## 2. 三个新增模块

### 模块 A · 功能介绍（轻 · 挂在状态 1 引入）
引入时不只是打招呼，而是给用户一张**「什么话 → 触发什么功能」对照表**，让用户知道怎么用：

| 用户这样说 | 触发 |
| --- | --- |
| 发**链接** / **截图** | 导入酒店（状态 3） |
| 「看结果」「好了」「没了」 | 出对比网页 H5（状态 4） |
| 「删掉 XX」「把贵的去掉」 | 删除酒店（状态 5） |
| 选哪个城市 | 设定通勤锚点 |

> ⚠️ **只引导链接 / 截图两种导入方式，不要让用户「发酒店名」**——酒店名歧义大、定位不准。
> 也支持用户在任何时候问「你能干嘛 / 帮助」时再调出这张表（闲聊里的一个意图）。

### 模块 B · 安全引导（轻 · 闲聊里的一个意图分支）
当用户点开我们发的 H5 链接，浏览器显示**「不可信 / 未备案」**，用户回来质疑「这安全吗 / 怎么打不开」时，Bot 安抚：

> 「链接是可信的！放心点开，我们正在提交备案，通过之后就能直接打开了。」

- 这是**被动触发**的意图（用户问才回），不是状态。
- ⚠️ **过渡方案**：备案通过后应能一键关掉这个分支。

### 模块 C · 智能删除（重 · 独立状态 5，见第 4 节）
用户说「把 XX 删了」时**不能直接删**，要走带二次确认的子状态机。详见第 4 节。

## 3. 状态 2（闲聊 idle）下的决策树

每收到一条消息，Bot 先做**语义判断（LLM）**，归入下面五个分支之一。

### 分支 A — 导入酒店意图
触发：发了**链接 / 截图**（不引导酒店名）。
```
是否已选城市？
├─ 否 → 引导先选城市（不进入导入）
└─ 是 → 进入【状态 3 导入】→ 解析存库 → 回 idle
```

### 分支 B — 完成导入意图
触发：如「好了」「看结果」「没了」。
```
判断 1：是否已选城市？  否 → 引导先选城市
判断 2：是否已有导入酒店？
  ├─ 否 → 告知还没酒店，建议发截图或链接
  └─ 是 → 进入【状态 4 交付】→ 发 H5 链接 → 回 idle
```

### 分支 C — 普通闲聊意图
```
以「向导人设」风趣回复 → 维持 idle
```

### 分支 D — 安全质疑意图（模块 B）
触发：用户质疑链接不可信 / 打不开。
```
回复「链接可信，正在备案」安抚文案 → 维持 idle
```

### 分支 E — 删除酒店意图（模块 C）
触发：用户表达想删除部分已导入酒店。
```
进入【状态 5 删除】
```

## 4. 状态 5 · 删除酒店（子状态机）

**进入条件**：在 idle 中，用户表达「删除部分已导入酒店」的意图。

**入口动作**：调用**匹配函数（LLM）**——把 `用户语义（"全部"/"贵的"/"汉庭的"…）+ 已导入酒店列表` 给 LLM，返回命中的 `x` 个酒店。按 `x` 分流。

```
进入状态 5
  │ 匹配函数(LLM): (用户语义, 已导入酒店) → 命中 x 个
  │
  ├─ x = 0 ──→ 【子状态 5.1：空匹配】
  │   回复："没匹配到酒店，还要删吗？换个角度描述一下？"
  │   等用户回应（语义判断 LLM）：
  │     ├─ 想继续      → 重新进入状态 5 入口（让用户重新描述）
  │     ├─ 不删了      → 返回 idle
  │     └─ 闲聊        → 计入「闲聊缓冲」(见下) 
  │
  └─ x > 0 ──→ 【子状态 5.2：确认删除】
      展示命中的 x 个酒店，问："是否删除这些？"
      等用户回应（子意图判断 LLM，5 选 1）：
        ├─ 「是」        → 移除这几个
        │                 → 回执："已移出 X 家，还剩 Y 家" → 返回 idle
        ├─ 「进一步需求」 → 调 refine：把[当前待删酒店 + 全部已导入酒店 + 用户新prompt]
        │                  给 LLM → 返回新的待删 list → 回到 5.2 重新确认
        ├─ 「重新来」     → 清空待删 → 回状态 5 入口，让用户重新描述
        ├─ 「不删了」     → 返回 idle
        └─ 「闲聊」       → 计入「闲聊缓冲」(见下)
```

### 闲聊缓冲（5.1 与 5.2 都有）
- 用户在删除流程中说与删除无关的闲聊话 → Bot 正常闲聊回复，但**累计计数**。
- **连续 3 轮纯闲聊** → 视为放弃删除，超时**恢复到 idle**。
- **任何删除相关的有效动作**（重新匹配 / refine / 确认）→ **闲聊计数清零**。

### 关键观察
- 状态 5 里有 **2 处 LLM 调用**：① 入口/refine 的**匹配函数**（语义 → 酒店子集）；② 子状态 5.2 的**子意图判断**（是 / 进一步需求 / 重新来 / 不删了 / 闲聊，5 选 1）。
- 整个流程要在多条消息间**保留上下文**：当前待删 list + 闲聊缓冲计数。

## 5. 逻辑关键点对齐

| 关键点 | 说明 |
| --- | --- |
| **非强制性** | 用户在「引入」后高度自由：可先闲聊，也可直接喂数据，不被流程绑死。 |
| **前置依赖** | 所有功能执行（导入、看结果）都**强依赖「已选城市」**——城市是地图上算通勤的锚点。 |
| **容错弥补** | 缺数据时 Bot 以**积极态度**指出缺失并引导补全，不冷冰冰报错。 |
| **删除必确认** | 删除是破坏性操作，**必须二次确认**，绝不静默删除。 |
| **匹配/判断皆 LLM** | 「语义 → 酒店子集」的匹配函数、各处意图判断，**都由 LLM 完成**，不写死规则。 |

## 6. 云端 AI 实现链路

收到消息后，Bot **不直接把整段对话丢给大模型**，而是先**判断意图（LLM）**拿到一个 `prompt_key`，用 key 取出对应 Prompt 模板，再把「Prompt + 上下文」一起发给 DeepSeek API。

```
收到消息
  → 当前状态？（idle / 删除子状态 / …）
  → 意图判断（LLM）→ 得到 prompt_key 或 子意图
  → 取出对应 Prompt 模板
  → 组装 [Prompt 模板 + 上下文(含待删list/缓冲计数等)]
  → 调用 DeepSeek API
  → 回复用户 + 更新状态
```

**判断模块是核心**：它把「自然语言意图」收敛成有限个 `prompt_key` / 子意图，每个对应一段可单独维护、可灰度调整的 Prompt，使大模型行为**可控、可测、可迭代**。

### 伪代码

```python
def handle_message(msg, ctx):
    if ctx.state == "delete":
        return handle_delete_state(msg, ctx)      # 状态5 子状态机
    intent = classify_intent(msg, ctx)            # LLM 语义判断 → 分支 A~E
    if intent == "delete":
        ctx.state = "delete"
        return enter_delete(msg, ctx)             # 入口匹配
    prompt = PROMPT_TABLE[intent_to_key(intent)]
    return deepseek(prompt=prompt, context=ctx)

def enter_delete(msg, ctx):
    hits = llm_match_hotels(msg, ctx.imported_hotels)   # 匹配函数(LLM)
    if not hits:
        ctx.substate = "5.1"; return ask_empty_match()   # 子状态5.1
    ctx.pending_delete = hits
    ctx.substate = "5.2"; return ask_confirm(hits)        # 子状态5.2
```

## 7. Prompt 表（草案 · 具体 Prompt 待填）

每个场景对应一个 `prompt_key` 与一段 Prompt 模板。匹配函数、意图判断函数也各是一段 Prompt。

| # | prompt_key（建议） | 场景 | 设计 / 理由 |
| --- | --- | --- | --- |
| 1 | `greeting_guide` | 问候 + 功能介绍（模块 A） | 打招呼并给出「语义→功能」对照表，教用户怎么用。 |
| 2 | `init_city_setup` | 初始化引导（选城市/景点） | 确保地图上有算通勤的锚点。 |
| 3 | `import_processing` | 导入实时反馈（处理中） | 体现「正在扫描/解析」的进度感。 |
| 4 | `import_result` | 解析结果反馈（含纠错） | 成功报喜；失败积极弥补并引导重发。 |
| 5 | `compare_nudge` | 对比决策触发（Nudge） | 数据足够时推用户去看网页。 |
| 6 | `query_detail` | 咨询地点/酒店详情 | 体现「整合者」专业。 |
| 7 | `h5_entry` | 返回规划网址（H5 入口） | 核心入口，文案要诱人。 |
| 8 | `boundary_paywall` | 能力边界 + 付费引导 | 「不能订房」+「付费开高级工具」连贯。 |
| 9 | `billing` | 计费（支付/到期） | 商业闭环。 |
| 10 | `chitchat_persona` | 向导式闲聊（含认错） | 强化人设，可靠但会犯错的亲切感。 |
| 11 | `share_feedback` | 分享与反馈引导 | 增加传播属性。 |
| 12 | `safety_reassure` | 安全引导（模块 B） | 安抚链接可信、正在备案；备案后可关。 |
| 13 | `intent_classify` | 闲聊意图判断（LLM 函数） | 把消息分到分支 A~E。 |
| 14 | `delete_match` | 删除匹配函数（LLM 函数） | 语义 + 已导入酒店 → 待删酒店子集。 |
| 15 | `delete_empty` | 删除·空匹配（子状态 5.1） | 命中 0 家，引导「换个角度描述 / 还删吗」。 |
| 16 | `delete_confirm` | 删除·确认（子状态 5.2） | 展示待删 list，问「是否删除这些」。 |
| 17 | `delete_subintent` | 删除子意图判断（LLM 函数） | 5 选 1：是/进一步需求/重新来/不删了/闲聊。 |
| 18 | `delete_done` | 删除·回执 | 「已移出 X 家，还剩 Y 家」。 |
