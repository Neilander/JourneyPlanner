# AI旅行地图 MVP 功能实现报告

## 第一部分：项目简述

### 1. 一句话定义

把用户想去的景点、候选酒店和通勤成本放到一张地图上，让用户快速判断住哪里最方便。

### 2. 项目定位

本项目不是完整的 AI 旅游规划工具，也不是 AI 攻略生成器。

MVP 阶段的核心定位是：

- 旅行住宿决策工具
- 景点-酒店关系可视化工具
- 通勤成本对比工具
- 酒店避雷辅助工具

核心问题是：

> 用户已经大概知道自己想去哪玩，但不知道住哪家酒店最方便。

### 3. MVP核心流程

用户使用路径如下：

```text
进入城市地图
↓
浏览预置景点 / 地铁线路
↓
在底部横向卡片中多选景点或线路
↓
导入候选酒店
↓
地图根据选择状态展示酒店位置价值
↓
排行榜显示各酒店平均通勤时间
↓
点击酒店查看详情与 AI 避雷信息
```

### 4. MVP核心模块

MVP 包含四个核心模块：

1. 城市景点地图
2. 底部横向选择卡片
3. 酒店通勤排行榜
4. 酒店详情与 AI 避雷

暂不做：

- AI 自动生成完整行程
- 推荐景点
- 酒店预订
- 酒店比价
- 社交功能
- 多城市复杂行程规划

## 第二部分：界面功能与信息设计

## 1. 城市地图页面

### 1.1 页面结构

页面上方为功能切换：

```text
地图 / 排行榜
```

页面主体为地图区域。

页面底部为横向滑动卡片栏，用于选择景点和地铁线路。

基础结构：

```text
顶部：地图 / 排行榜切换

中间：地图或排行榜内容

底部：横向滑动选择卡片
```

---

## 2. 底部横向选择卡片

### 2.1 卡片内容

底部卡片不只包含景点，也包含地铁线路。

示例：

```text
[大唐不夜城]
[钟楼]
[兵马俑]
[陕西历史博物馆]
[地铁2号线]
[地铁4号线]
```

### 2.2 卡片状态

每张卡片支持：

- 未选中
- 已选中

用户可以多选，也可以取消选择。

### 2.3 卡片作用

底部卡片决定地图和排行榜的计算对象。

用户选择的景点 / 地铁线路，会影响：

- 地图上的酒店颜色
- 排行榜中的平均通勤时间
- 酒店详情中的到达时间
- 通勤方式构成比例

---

## 3. 地图模块

地图模块根据用户选择状态分为三种模式：

---

### 3.1 无选择状态

用户尚未选择任何景点或地铁线路。

地图只显示导入的住宿酒店。

酒店视觉规则：

```text
评分越高，酒店点越大
```

此状态下地图主要回答：

> 我现在有哪些候选酒店，它们大概分布在哪里？

不显示过多景点、餐厅、商圈等信息，避免地图变乱。

---

### 3.2 单选状态

用户只选择了一个景点或一条地铁线路。

#### 选择景点时

地图只显示：

```text
选中的景点
+
所有候选酒店
```

酒店视觉规则：

```text
评分越高，点越大
距离景点越近，颜色越深
```

此状态下地图回答：

> 哪些酒店离这个景点最近？

#### 选择地铁线路时

地图只显示：

```text
选中的地铁线路
+
所有候选酒店
```

酒店视觉规则：

```text
评分越高，点越大
距离线路越近，颜色越深
```

此状态下地图回答：

> 哪些酒店更靠近这条地铁线路？

---

### 3.3 多选状态

用户选择了多个景点 / 地铁线路。

地图显示：

```text
已选景点
+
已选地铁线路
+
所有候选酒店
```

酒店视觉规则：

```text
评分越高，点越大
综合平均通勤越优，颜色越深
```

这里的“综合平均通勤”采用平均时间计算。

例如用户选择了 3 个景点：

```text
酒店A 到景点1：10分钟
酒店A 到景点2：20分钟
酒店A 到景点3：30分钟

酒店A 平均通勤 = 20分钟
```

此状态下地图回答：

> 从整体位置来看，哪些酒店对我选中的这些地方更方便？

---

## 4. 排行榜模块

排行榜模块与地图使用同一套数据。

地图用颜色表达通勤优劣，排行榜用排序表达通勤优劣。

---

### 4.1 无选择状态

如果用户没有选择任何景点或线路，排行榜不显示具体酒店排序。

显示引导文案：

```text
选择想去的景点或地铁线路，查看酒店通勤排行
```

---

### 4.2 已选择状态

当用户选择一个或多个景点 / 线路后，排行榜开始显示酒店排序。

排序规则：

```text
平均通勤时间从低到高
```

酒店卡片示例：

```text
西安钟楼亚朵

平均通勤：18分钟

🚶 15%  🚇 70%  🚌 10%  🚕 5%
```

另一个示例：

```text
全季酒店

平均通勤：22分钟

🚶 10%  🚇 30%  🚌 20%  🚕 40%
```

---

### 4.3 通勤方式构成条

每张酒店卡片需要显示通勤方式构成。

构成包括：

- 步行
- 地铁
- 公交
- 打车

这个比例不是用户手动填的，而是系统根据路线规划结果统计出来的。

例如用户选择了 4 个景点，系统分别计算酒店到每个景点的路线，然后汇总路线中的交通方式占比。

作用是让用户知道：

> 这个酒店看起来平均时间短，是因为真的地铁方便，还是因为大量依赖打车？

---

## 5. 通勤设置

排行榜页面或地图页面提供一个设置入口。

例如：

```text
⚙ 通勤偏好
```

### 5.1 时间最短

用户选择“时间最短”时，系统默认接受所有交通方式，包括：

- 步行
- 地铁
- 公交
- 打车

系统对每个酒店到每个景点计算最快路线。

如果打车最快，就采用打车时间。

### 5.2 公共交通优先

用户选择“公共交通优先”时，可以勾选允许的公共交通方式。

例如：

```text
☑ 地铁
☑ 公交
☑ 步行接驳
```

系统在用户允许的交通方式内，选择最快路线。

### 5.3 设置对结果的影响

通勤设置会影响：

- 地图中酒店颜色深浅
- 排行榜排序
- 平均通勤时间
- 通勤方式构成条
- 酒店详情中的到达时间

---

## 6. 酒店导入方式

MVP 需要准备多套酒店导入方案，避免单一路径失败。

---

### 6.1 方案A：携程 / 美团分享卡片导入

用户从携程或美团分享酒店卡片给企业微信客服。

后端尝试解析：

- 酒店名称
- 酒店地址
- 酒店评分
- 原始链接
- 经纬度

如果卡片里没有经纬度，则通过酒店名称或地址调用高德 POI 搜索 / 地理编码获取坐标。

这是最佳体验，但稳定性不一定最高。

---

### 6.2 方案B：截图识别导入

用户发送酒店截图。

系统通过 OCR / 多模态模型识别：

- 酒店名称
- 地址
- 平台评分
- 价格
- 评论摘要

然后用酒店名称或地址调用高德 POI 搜索获取经纬度。

这个方案适合作为容错路径。

---

### 6.3 方案C：搜索酒店导入

用户点击：

```text
+ 添加酒店
```

输入酒店名称。

例如：

```text
西安钟楼亚朵
```

系统调用高德 POI 搜索，返回候选结果。

用户确认后导入。

这是最稳定的保底路径。

---

### 6.4 方案D：预置酒店库导入

系统提前为热门城市准备一批热门酒店数据。

用户可以从预置酒店列表中直接导入。

适合 MVP 初期让页面一打开就有内容，不至于空白。

---

## 7. 酒店详情页

用户点击地图上的酒店点或排行榜中的酒店卡片，进入酒店详情。

详情页包含三类信息。

### 7.1 基础信息

包括：

- 酒店名称
- 平台评分
- 地址
- 来源平台
- 原始链接

### 7.2 通勤信息

显示该酒店到用户已选择景点 / 线路的时间。

示例：

```text
到大唐不夜城：12分钟
到钟楼：18分钟
到兵马俑：35分钟
```

### 7.3 AI避雷信息

如果系统获得了评论文本或截图评论，可以生成 AI 避雷报告。

内容包括：

- 风险等级
- 核心雷点
- 适合人群
- 不适合人群
- 简短总结

示例：

```text
风险等级：黄色

核心雷点：
- 隔音一般
- 电梯等待时间长
- 设施偏旧

适合：
- 预算有限
- 白天行程多
- 只住一两晚

不适合：
- 睡眠浅
- 带老人小孩
- 对酒店硬件要求高
```

AI 避雷是辅助决策，不是主界面的第一优先级。

---

## 8. 用户使用流程 Case

### Case 1：用户已有候选酒店

```text
用户进入城市页
↓
导入 2-5 个候选酒店
↓
在底部卡片选择想去的景点
↓
切换排行榜
↓
查看平均通勤时间
↓
点开酒店详情查看避雷信息
↓
决定住哪家
```

适合用户已经在携程 / 美团 / 小红书看了一批酒店的场景。

---

### Case 2：用户还没有酒店，只想先看位置

```text
用户进入城市页
↓
系统显示预置热门景点和地铁线路
↓
用户选择想去的景点
↓
查看地图上的空间关系
↓
再搜索或导入酒店
↓
用排行榜比较酒店
```

适合用户还在早期规划阶段。

---

### Case 3：用户通过截图导入酒店

```text
用户发送酒店截图给企业微信客服
↓
系统识别酒店名称、地址、评分
↓
调用高德 POI 搜索获取坐标
↓
把酒店加入当前城市地图
↓
用户选择景点
↓
查看通勤排行
```

适合携程卡片解析失败，或者用户不方便分享链接的场景。

---

### Case 4：用户只想看公共交通方便程度

```text
用户选择多个景点
↓
打开通勤偏好
↓
选择公共交通优先
↓
勾选地铁 / 公交 / 步行接驳
↓
系统重新计算排行榜
↓
用户查看哪些酒店不依赖打车
```

适合不想频繁打车、预算敏感、重视地铁便利性的用户。

## 第三部分：需要准备的工作

## 1. 城市数据准备

MVP 需要先选择少数热门城市，不建议一开始做太多。

建议第一批城市：

```text
西安
上海
北京
成都
大阪
东京
```

也可以先只做 1-3 个城市。

每个城市需要准备：

- 城市名称
- 城市中心点坐标
- 地图默认缩放级别
- 行政区划代码
- 默认展示范围

---

## 2. 景点数据准备

每个城市需要预置 20-50 个热门景点。

每个景点需要字段：

```text
id
city_id
name
latitude
longitude
category
tags
description
popularity
```

示例：

```text
大唐不夜城
纬度
经度
类型：夜景 / 商业街 / 免费
标签：晚上适合、拍照、亲子
```

景点数据来源可以包括：

- 高德 POI 搜索
- 人工整理
- 小红书 / 携程 / 马蜂窝热门榜单人工参考
- 城市旅游官网公开信息

---

## 3. 地铁线路数据准备

每个城市需要准备主要地铁线路。

每条线路需要字段：

```text
id
city_id
line_name
line_color
stations
polyline
```

其中：

- stations：站点列表
- polyline：线路在地图上的折线路径

地铁线路用于：

- 地图显示
- 用户底部卡片选择
- 判断酒店距离线路远近
- 公共交通便利性分析

数据来源：

- 高德地图地铁数据
- 开放地铁线路数据
- 人工整理 GeoJSON
- 城市轨道交通公开线路图

MVP 初期可以不追求完整地铁网络，优先准备最常用的几条线路。

---

## 4. 酒店数据准备

酒店来源分为两类：

### 4.1 用户导入酒店

通过：

- 携程 / 美团分享
- 截图识别
- 手动搜索

导入。

需要字段：

```text
id
user_id
city_id
name
address
latitude
longitude
rating
source
source_url
created_at
```

### 4.2 系统预置酒店

为了避免初始页面空白，可以预置部分热门酒店。

每个城市可以准备：

```text
30-100 家热门酒店
```

预置酒店字段：

```text
id
city_id
name
address
latitude
longitude
rating
price_level
source
```

预置酒店的作用：

- 让用户一进页面就有内容
- 用于 Demo 展示
- 作为搜索导入失败时的补充

---

## 5. 路线规划数据准备

系统需要调用地图路线规划 API。

可选服务：

- 高德 Web 服务 API
- Mapbox Directions API
- Google Maps Directions API

国内 MVP 优先高德。

需要支持：

- 酒店到景点的步行时间
- 酒店到景点的公共交通时间
- 酒店到景点的打车 / 驾车时间
- 酒店到地铁线路的距离
- 酒店到地铁站的距离

路线计算结果需要缓存。

缓存字段：

```text
origin_id
destination_id
mode
duration
distance
segments
updated_at
```

这样可以减少 API 成本，也能提升页面响应速度。

---

## 6. AI避雷数据准备

AI避雷依赖酒店评论。

评论来源可以是：

- 用户粘贴评论
- 用户上传评论截图
- 用户分享平台链接后成功解析评论
- 人工整理评论样本

AI分析结果字段：

```text
hotel_id
risk_level
scores
red_flags
best_for
avoid_if
summary
evidence_quotes
created_at
```

其中：

```text
risk_level = green / yellow / red
```

AI 避雷输出应结构化，方便前端展示和排序。

---

## 7. 企业微信 / 微信客服准备

需要确认实际入口采用哪一种：

- 企业微信外部联系人
- 微信客服
- 企业微信客服坐席
- 公众号菜单跳转

需要测试：

1. 用户能否直接发送携程 / 美团卡片
2. 后端能收到什么格式的消息
3. 是否能拿到 title、url、appid、pagepath
4. 图片是否能稳定接收
5. 是否能回复 H5 链接
6. 微信内置浏览器是否能打开结果页

这是上线前必须做的基础测试。

---

## 8. H5页面部署准备

H5 页面需要有可访问 URL。

推荐准备：

- Cloudflare Pages
- Vercel
- 腾讯云 CloudBase
- 腾讯云 EdgeOne Pages
- 阿里云 OSS 静态托管

MVP 阶段重点是：

```text
企业微信机器人能返回一个链接
用户点击链接能在微信内打开地图页面
```

---

## 9. 数据库准备

MVP 可以先用 SQLite，后续迁移 PostgreSQL。

核心表包括：

```text
users
cities
locations
hotels
attractions
subway_lines
user_selected_items
route_cache
ai_hotel_reports
```

建议统一抽象 Location：

```text
Location
- hotel
- attraction
- subway_station
- custom_place
```

因为系统底层真正需要的是：

```text
名称
类型
经纬度
城市
```

## 10. 开发优先级

### 第一优先级

必须完成：

- 城市地图页
- 预置景点
- 底部横向多选卡片
- 酒店手动搜索导入
- 酒店点显示
- 地图三状态
- 排行榜平均通勤
- 通勤偏好设置
- H5 分享链接

### 第二优先级

尽量完成：

- 携程 / 美团分享解析
- 酒店截图识别
- AI避雷报告
- 通勤方式构成条
- 酒店详情页

### 第三优先级

后续优化：

- 更多城市
- 更多地铁线路
- 评论证据引用
- 多人分享结果页
- 酒店收藏
- 自动识别用户行程偏好

---

## 11. 最小可上线版本

如果只做最小版本，建议范围是：

```text
1个城市
20个预置景点
3-5条地铁线路
酒店手动搜索导入
地图三状态
排行榜平均通勤
H5链接分享
```

这个版本已经可以验证核心价值：

> 用户是否愿意通过选择景点和导入酒店，来比较住宿位置是否方便。

如果这个价值成立，再继续增强 AI避雷、截图识别和携程卡片解析。
